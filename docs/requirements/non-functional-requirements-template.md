# 非機能要件定義書テンプレート（小規模・低予算向け）

## 1. 信頼性要件

### 1.1 可用性
*   システム稼働時間： 平日9:00〜18:00を想定（一般的な業務時間外は停止・保守許容）
*   目標復旧時間 (RTO)： 2週間以内（障害発生からシステム復旧までの時間）
*   目標復旧時点 (RPO)： 2週間以内（データ損失許容範囲。前日までのデータは復旧可能
※しかしながら可能な範囲で素早く対応する為、基本的には5日程度で復旧見込み

### 1.2 データの保全
*   データのバックアップは実施しない

## 2. セキュリティ要件

### 2.1 認証・認可
*   ユーザー認証はID/パスワードによる基本的な認証とする。

### 2.2 データ保護
*   個人情報や機密情報はサードパーティ製のサービスで管理する。
*   インターネット経由でのアクセスがある場合は、HTTPSによる通信暗号化を適用する。

## 3. 保守性要件

### 3.1 運用性
*   エラー発生時にシステム管理者がアプリの状況を把握できる程度のログを出力する。
*   簡単な運用マニュアルはお客様の方で準備する。

### 3.2 拡張性
*   将来的な機能追加やユーザー数増加に対応する明確な計画は設けないが、一般的な開発手法（モジュール化など）を用いる。
*   しかしながら、システムの規模が大きくなる場合に備えた設計を行う。

## 4. 運用要件

### 4.1 バックアップ・リカバリ
*   週に一度、自動または手動でのバックアップを実施する。
*   バックアップからの復旧手順を準備する。

## 5. 環境要件

### 5.1 稼働環境
*   クラウドサービス（SaaS/PaaS）の利用を検討し、インフラ構築・管理コストを抑える。
*   または、既存のサーバー環境を利用する。

### 5.2 開発環境
*   開発言語やフレームワークは一般的なものを採用し、開発者の確保や情報収集の容易性を考慮する。

## 6. その他の要件

### 6.1 法令・規制遵守
*   システムが関連する可能性のある最低限の法令・規制（個人情報保護法など）を確認し、遵守に努める。

### 6.2 品質
*   最低限のテスト（単体テスト、結合テスト）を実施し、基本的な動作保証を行う。
